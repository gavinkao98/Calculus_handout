裁決稿：scene-level TTS＋forced alignment 的 production 化設計

2026-07-05・分支 video/template-redesign-navy-spine・狀態：等使用者過目（過目後出實作計畫，才動 code）
前情：討論總結・ 三場景實測＋stable-ts 複驗（RESULTS）・ 進度錨 video/REBUILD_STATUS.md 2026-07-05 節

0. TL;DR 與已定案前提

已定案（2026-07-05 使用者同意，不在本稿重議）：
  • 走 hybrid：storyboard {show}／beat 契約不動；TTS 合成單位升到 content scene；per-scene validation 決定每個 scene 用 scene-level 或回退 beat-level。
  • 計時來源＝stable-ts transcript-constrained forced alignment（三類場景 197/197、209/209、99/99 全過，derivation 場景 beat 邊界誤差 0.06s）；whisper-timestamped 自由 ASR 降級為 QA 探針（抓「TTS 沒唸／唸錯」）。
  • 不做「全節／全片一條音訊」；不做固定 2–3 beat chunk（chunk 只當 fallback、切句界）。

本稿把上述方向落成可實作的契約：manifest schema 2（新 narration_mode: "scene_aligned"）、新模組 pipeline/scene_align.py、validation gates（初始門檻含依據）、fallback ladder、reuse／freshness 規則、7 個消費端的遷移点。核心設計選擇是：scene_aligned 條目保持與 beats 模式相同的 beats[] 形狀＋scene 級 audio_file——7 個既有消費端中，4 處只需把條件由 == "beats" 放寬為「兩模式皆可」、2 處零改動，只有 _validate_reuse_manifest 需要真正的新分支，改動面極小。

1. 目標／非目標

目標非目標（明確不做）
  • 聲線連貫：一個 content scene 一次合成（beat 接縫的 prosody 斷裂與 drift 消失）。
  • API call 數降一個量級（§3.1：99 beats → 21 scenes）。
  • reveal 時序由 FA 回推、逐 scene 驗證，未通過自動走 fallback，永遠有可出片的路。
  • beat-level 路線原樣保留（pre-lock 迭代、mock、fallback 終點）。
  • 不動 storyboard 格式、不動 {show} 語義、不動 NFA／lock 流程（scene transcript＝同一批 beat 文字的 join，NFA 審的源文字不變）。
  • 不動 intro／outro／divider（kind != content，house audio 路線照舊）。
  • 不做 voice-design 克隆聲線；voice 仍是 tts.py --voice 參數（與本設計正交）。
  • 不在 code 裡強制 lock 狀態檢查（lock 是 RUNBOOK 程序紀律，見 §10）。

2. 資料流總覽

storyboard (say + {show})                        ↓ 每個 content scene
   └─ scene plan：transcript ＝ join(beat texts)；每 beat 的 token 範圍 [word_start, word_end)
        └─ MiMo TTS（1 call／scene，計費）→ scene WAV
             └─ stable-ts align(WAV, transcript)（本機 ~3s/scene）
                  └─ explode 成 plan tokenization → 逐字驗 index（不符＝abort）
                       └─ validation gates（§5）＋ ASR QA 探針（§6）
                            ├─ PASS → aligned beats（每 beat start/dur）→ manifest scene_aligned
                            └─ FAIL → fallback ladder（§7）→ 最終 beat-level（既有路線）
manifest（schema 2）→ make.py --reuse-audio → render（beat durations 注入）→ compose（scene WAV 直接 mux）

與實驗線的對應：plan＝prepare_scene.py、對位＝run_stable_ts_align.py、映射＝map_alignment_to_beats.py、render 注入＝render_aligned_scene.py 已各自驗證過；production 化＝把這四步的核心搬進 pipeline/scene_align.py 並接進 tts.py／make.py。實驗腳本不動（保留為 lab bench）。

Freshness／reuse 規則（兩層獨立，這是 scene-level 的重要紅利）：

變更TTS（計費）對位＋驗證（本機免費）判定依據
storyboard 該 scene 任一 beat 文字改了重合成整個 scene重跑scene_text_hash＋per-beat text_hash 不符
reveal id／beat 數改了（文字沒改）沿用 WAV重跑映射＋驗證per-beat reveal／beat_count 不符、scene_text_hash 相符
aligner 升版／換 model／調參沿用 WAV重跑alignment.aligner 與現行設定不符
backend／model／voice／style 改了重合成重跑頂層四欄不符（沿用既有 tts.py --reuse-existing 紀律）
WAV 實際時長 ≠ manifest audio_seconds視為 stale，重合成重跑檔案被動過／半寫入（沿用 make.py 既有檢查）
  • 原子寫入：words／aligned／manifest 一律寫 .tmp 再 rename，杜絕「舊 words.json 配新 WAV」。
  • fallback 產物驗證通過才覆蓋既有 artifacts；覆蓋前舊檔不刪。

5. Validation gates（boundary-focused）

原則（pilot 實測直接教的）：reveal 時序只取決於「每個 beat 第一個字」的 timestamp——閘門盯邊界字，beat 內部的低機率字只 WARN。三場景的內部低機率字（O A 唸字母、one half）全是無害噪音，若拿「任何低機率 run 就 fail」會誤殺兩個完美場景。scene 平均 confidence 無鑑別力（壞場 0.776 vs 好場 0.833），不當閘。

檢查等級初始門檻依據（pilot 數據）
stable-ts failure_threshold abort（對不上 transcript 的字超比例）FAIL0.2音檔與 transcript 嚴重不符（合成錯檔、髒 WAV）時快速止血
explode 後 token 序列 ≠ plan tokens（逐字）FAIL嚴格相等by construction 不該發生；發生＝index 契約破裂，禁止產出時間軸
word timestamps 非單調／重疊FAIL任一aligner 異常輸出
零時長字：連續 ≥2 個，或總數佔比 >5%FAIL如左pilot 僅見孤立 1 個（'a'），孤立屬無害（不動邊界）→ 孤立僅 WARN
beat 邊界字（beat 2..N 首字）機率過低FAIL<0.15（0.15–0.35 WARN）觀測到的正常邊界字最低 0.53（Look）；遠低於此＝邊界不可信
beat 邊界落在多 token 字內（timestamp 是內插值）WARN＋修正邊界 snap 到父字 start三場景 0 例（全庫僅 1 個多 token 字）；snap 使 reveal 略早——「先亮再講」與契約同向，晚才有害
scene 內部無法解釋的字間空洞（不在句界）FAIL>2.0s合法句間停頓最大 0.89s；大空洞＝TTS 出怪聲／重複的嫌疑位
beat 時長 vs 字元占比（末 beat 以「末字 end」計，排除尾靜音）WARN／FAIL比值出 [0.6,1.6] WARN；出 [0.4,2.2] FAILpilot 正常比值 0.9–1.3；derivation 事故時壞 beat 曾灌到吸走鄰 beat（舊法 16.9s）
beat 時長 <0.05sWARN既有（沿用 mapper）短 reveal 由 _warn_short_beats 的動畫墊時機制接手
內部低機率 run（≥2 字 <0.5）／prob_mean <0.7WARN如左記錄供人看，不擋（見上原則）

門檻全部集中在 scene_align.GATES 一個 dict（值＋註解），標注「以 3-scene pilot 校準的初值，首個真 deck rollout 後回校」。任何 FAIL → 該 scene 進 §7 ladder；WARN 進 manifest validation.warnings＋console。

6. ASR QA 探針（whisper-timestamped 的新職務）

transcript-constrained FA 結構上「找不到 transcript 沒有的東西」——TTS 自己漏唸／重複／亂唸，FA 只會表現成低分或怪 gap，說不出是什麼。ASR 探針補這一刀：對 scene WAV 跑自由 ASR，正規化後與 transcript 做 token diff（difflib.SequenceMatcher），用 FA 的 word timestamps 把 diff 定位到 beat。

情形判定動作
ASR 缺漏／插入叢集（≥3 tokens）且同區 FA 低機率或 gate 異常FAIL（suspect TTS misspeak）進 §7 ladder（通常第 2 級重合成就解）
ASR 缺漏但 FA 該區高分、靜音結構正常INFO（ASR artifact）記錄即可（derivation 事故正是此類——ASR 跳字、音檔無辜）
兩組 ASR 設定（greedy vs beam）對同一 diff 意見一致升級 confirmed提高上表判定的信心；insertion 與 deletion 同權重

預設：真合成路徑開（本機 CPU，~20–60s/scene，只在合成後跑一次）；--skip-qa 可關；mock 路不跑。

7. Fallback ladder（per-scene、lazy、驗過才覆蓋）

  1. 換 small.en 重對位（本機、零 API）：當 arbiter 用——兩個 model 邊界一致＝FA 可信度升；仍 FAIL 才往下。
  2. 重合成該 scene 一次（1 API call）：TTS 非決定性，重抽常直接解（尤其 suspect misspeak）。
  3. 句界 chunk：transcript 按句界切 2–3 段（切點選最靠近 beat 邊界的句號），每 chunk 一次 TTS → per-chunk FA → word 時間軸加 offset 合併 → WAV concat 成單一 scene 音檔 → 重跑 gates。產物仍是 scene_aligned（alignment.chunks 記錄切法），下游不可見。
  4. beat-level TTS（既有路線，該 scene 寫回 narration_mode: "beats"）：永遠可用的終點。
  • lazy：beat fallback 音訊只在走到第 4 級才合成，絕不預付雙倍 TTS。
  • 每一步寫進 fallback_history（步驟＋觸發原因＋結果），事後可稽。
  • 第 2、3 級是計費動作。CLAUDE.md 明定「重跑需重新徵同意」，所以徵同意的聲明範圍要一開始就把 retry 預算列進去：每次 scene-level 合成的報價一律寫明「N scenes ＋ fallback 第 2/3 級每 scene 至多 2 次額外 call」，核准即涵蓋；超出此預算（例如同一 scene 三修不過）停下來回報。整章重跑仍另行報價。

10. Rollout 與工作流

  • lock 紀律（程序，不進 code）：scene-level 真合成只在 narration lock＋NFA 之後（RUNBOOK 更新步驟）；lock 前一律 make.py --backend mock（beats，零計費）。「改一個字→整 scene 重合成」的 blast radius 由此吃掉。
  • batch 1：--unit auto 對 template ∈ {definition_math, graph, callout, recap_cards} 走 scene；其餘（含 derivation／theorem_proof）暫走 beats。allowlist＝tts.py 一個常數。
  • batch 2：首個真 deck 跑完、gate 行為與門檻回校後，把 derivation／theorem_proof 加進 allowlist（pilot 已證明 FA 過得了，保守只是 rollout 順序）。終態＝全 content template 走 scene、validation 當唯一裁判。
  • 驗收（goal-driven）：① selftest 全綠＋mock dry-run §3.1 全 deck render／sync／compose 綠（離線零計費）→ ② §3.1 真 scene-level 合成 21 scenes（屆時依 CLAUDE.md 報價徵同意）→ 與既有 beat-level MiMo 成片 A/B（聲線連貫、reveal 同步、VLM 抽幀）。

11. 測試計畫（先寫測試，離線可全跑）

  • selftest（不需 whisper model，全吃合成 fixture）：explode／tokenize 不變量（對全部現有 storyboard 的 say 文字：explode(合成 word dicts)＝WORD_RE(transcript)，含 sum-to-product、--、縮寫）；映射邊界政策（首字 onset／末 beat 到檔尾／pause 歸前 beat）；gates 逐條 RED→GREEN（非單調、零時長 run、大 gap、邊界低機率、char-share 離群、multi-token snap）；QA diff 分級三情形；manifest v2 round-trip＋freshness 矩陣每格一個 case。
  • 整合（離線）：mock scene WAV（靜音＋linear words，實驗線 --linear-from-audio 已有此模式）跑通 tts.py scene 路徑→make.py render/compose；fallback ladder 用注入的 gate 失敗驅動到第 3、4 級（chunk 合併與 beats 回退各驗一次）。
  • 真資料回歸：對磁碟上三個實驗 scene 的 WAV 重跑 production 模組，結果須與實驗線輸出一致（時間軸 bit-identical）。

12. 風險與緩解

風險緩解
stable-ts 上游已封存（2026-05-30）釘 2.19.1（已驗證組合記於 ENVIRONMENT.md ⑤c）；備援＝torchaudio CTC forced alignment（依賴已在環境）；scene_align.py 把 aligner 呼叫隔離在單一函式後面，換 aligner 不動 gates／映射
FA 局部漂移（如 beat_05 案例若反過來）QA 探針交叉＋small.en arbiter＋邊界字機率閘；仲裁工具（ffmpeg silencedetect 比對停頓結構）已驗證可用，留在 RESULTS 供人工複核

13. 已否決方案（決策紀錄）

  • 全節／全片單一音訊：blast radius 與定位成本不成比例；scene 是天然 hard sync point。
  • 固定每 2–3 beat chunk 當預設：重新引入 prosody 接縫（正是要解的問題）；built-in voice 跨 call 本就穩，chunk 只值得當 fallback、且切句界。
  • 自由 ASR（whisper-timestamped）當計時來源：對重複公式片語會 decoder 跳字（實測 -12 字→後續 beat 全錯位 4.6–7.0s），連當錨點都不可靠（beat_05 案例）；數學推導天生觸發此機制，換大 model 治標。
  • 全域字數 delta ≤5% 當主閘：index 對位下 delta≠0 即已壞；且 12/197=6.1% 剛好觸發、scene 再長就放行同級災難。被逐字 index 驗證＋boundary gates 取代。
  • scene 平均 confidence 當閘：壞場 0.776 vs 好場 0.833，無鑑別力；平均稀釋局部災難。

14. 已取的預設值（不需裁決，但都可改）
